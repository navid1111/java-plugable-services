Ten services. One gateway. No shared tables.

A microservices backend where Kong verifies every request at the edge and each service is a self-contained plug kit — its own database, its own routes, wired together by events instead of foreign keys. Snap a service onto the gateway or pull it off without touching the rest.

The edge

Every request enters through Kong

Kong owns the public port, verifies the JWT, and forwards to the owning service on its prefix. /auth is public because it mints the tokens; everything else requires a bearer JWT.

How it holds together

Six patterns every service follows

The same contract repeats across the platform, which is what makes services interchangeable and independently deployable.

EDGE AUTH

JWT verified at the gateway

Kong checks the token once at the edge; services trust the verified identity. Signing secret and issuer are shared with Kong's JWT credential.

ISOLATION

Database per service

No shared tables. Data crosses a boundary only as an event or an authenticated HTTP export — never a direct read of another service's schema.

RELIABILITY

Transactional outbox

The domain row and its event commit in one transaction; a scheduled relay publishes to RabbitMQ and marks it sent. At-least-once, no dual-write race.

INTERNAL

Workload JWT plane

Service-to-service calls to /internal/* carry a separate short-lived workload token with a trusted-caller list — distinct from end-user JWTs.

READS

Denormalized identity

Services store userId + username inline, so reads never call auth synchronously. An identity backfill reconciles gaps from the auth export.

PACKAGING

Plug-kit per service

Each service ships a plug/ dir — a compose fragment, kong-setup.sh, and smoke.sh — so it plugs onto the gateway on its own.

The backbone

Services talk over one event exchange

A single durable topic exchange carries every domain event; the routing key is the event type. Each consumer binds only the keys it needs, with a dead-letter path for messages that fail after retry.

Run it

Up in three steps

Services turn on à la carte with compose profiles; RabbitMQ joins any event-driven profile automatically. Register each one at the gateway once Kong is healthy.

# 1 · secrets
echo "JWT_SECRET=change-me-…-min-32-bytes" > .env
echo "JWT_ISSUER=springboot-auth" >> .env

# 2 · start the profiles you want (additive)
docker compose --profile auth     up --build -d
docker compose --profile tweeter  up --build -d
docker compose --profile media    up --build -d

# 3 · register services at the gateway
./kong/setup-core.sh       # jwt issuer + auth (first)
./kong/setup-tweeter.sh    # /posts
./kong/setup-media.sh      # /media

# verify a full flow through Kong
./auth-service/plug/smoke.sh
